# 从 Demo 到可试点内部工作台

## 目标判断

当前版本已经是一个可讲清楚业务价值的高质量原型，但距离可试点内部工作台还差一层“队列运营能力”和一层“系统治理能力”。

最小升级原则：

- 不扩成重后台
- 不做多角色权限系统全量版
- 不追求复杂工作流编排
- 只补足试点阶段真正会被用到的能力

## 必须做

### 1. 队列化处理

- 负责人持久化，不再只是展示静态 owner
- 支持在详情页转派负责人
- 每个案例有 `due_at` 与 `sla_status`
- 队列支持按逾期查看
- 审计时间线保留分派记录

为什么优先：

没有负责人和 SLA，就还只是“案例展示器”，不是“运营工作台”。

### 2. 异常自动入库

- 从 seed / mock 数据自动生成异常案例
- 支持按计划任务或接口触发入库
- 避免必须手动创建案例才能演示流程

为什么优先：

真正的异常系统应当是“发现问题后进入队列”，而不是“人工先创建问题”。

### 3. 失败与补证闭环

- Dify 失败要有明确失败状态
- 证据不足要进入待补证，而不是只在文案里提示
- 补证后支持重新分析

为什么优先：

试点阶段最容易暴露的问题不是预测不准，而是数据不全和流程卡住。

## 应该做

### 4. 规则治理最小版

- 规则配置结构化
- 规则版本号写入审计记录
- 每条规则命中都能回溯到证据字段

### 5. 队列视角增强

- 支持按负责人、风险等级、SLA 状态筛选
- 支持保存常用视图
- 支持最近处理、待我处理两个视角

### 6. 运营度量

- 逾期数量
- 平均处理时长
- 各异常类型分布
- AI 分析失败率 / 人工兜底率

## 可以延后

### 7. 权限与组织模型

- 登录鉴权
- 多角色权限
- 组织树与分组策略

### 8. 生产级集成

- 真实 ERP / WMS / 财务系统接入
- Webhook / 定时扫描入库
- 生产写回保护

### 9. 规则中心与后台配置

- 可视化规则编辑
- SOP 配置中心
- Prompt / Workflow 版本管理

## 当前迭代落地范围

本轮先做“必须做”的第一批：

- 负责人持久化
- SLA 截止时间与逾期状态
- 队列按逾期筛看
- 详情页转派负责人
- 时间线保留分派记录

这批改完后，项目会从“可演示案例台”更接近“可试点运营台”。
